# 리서치 프로젝트 CLAUDE.md

이 파일은 프로젝트 루트에 두는 project-level CLAUDE.md 다. 프로젝트 어디서 작업하든 자동 로드된다. 리서치 관련 룰은 리서치 관련 트리거 (준비·실행) 가 잡힐 때만 발동하고, 이 프로젝트의 다른 작업 (LexBrief 본체 기능 등) 에는 영향을 주지 않는다. user-level CLAUDE.md 의 룰 위에 리서치 작업 특화 룰이 추가된다.

## Tone & Style

- **존댓말 사용** — 반말 금지. 캐주얼하되 존댓말로.
- 간결하게 핵심만. 불필요한 설명 X.

## 🔒 리서치 배포 워크플로우 (필수 순서)

리서치를 인터넷에 배포할 때는 반드시 다음 5단계를 **순서대로** 수행:

1. **research 폴더에 원본 markdown 작성** — `research/YYYY-MM-DD-{slug}/result.md` 형식.
2. **HTML 빌드** — `build/{slug}/index.html` 로 리서치 폴더 안 markdown 을 HTML 로 변환. 정적 사이트 배포용.
3. **이 저장소 안 `research/{slug}/` 하위 경로에 build 결과 추가** — 프로젝트 루트의 `index.html` 상단 Research 메뉴로 이어지는 `research/index.html` 목록에 새 리서치 링크 한 줄 추가.
4. **GitHub 에 commit + push** — Claude 에게 "GitHub 에 업데이트 올려줘" 한 줄 부탁하면 자동. Render 가 1~2분 안에 새 리서치 페이지를 같은 URL 하위 경로에 자동 배포.
5. **본인 스마트폰에서 새 리서치 URL 확인** — 정상 렌더링 확인.

### 룰 위반 시 발생하는 문제

- 원본 markdown 없이 HTML 만 있으면 → 나중에 수정할 때 HTML 직접 편집 또는 reverse engineering 필요 → 시간 낭비.
- markdown 은 리서치의 source of truth. HTML 은 그 변환 결과일 뿐.

### 매 배포마다 체크

- [ ] 원본 markdown 있는지
- [ ] HTML 빌드 결과 build/ 폴더에 있는지
- [ ] `research/{slug}/` 에 복사 + `research/index.html` 목록 업데이트 + push 했는지

## 🔒 신규 리서치 폴더 생성 규칙 (필수)

새로운 리서치를 시작해서 폴더 (예: `YYYY-MM-DD-주제/`) 를 생성할 때는 **반드시** 아래를 **먼저 질문**:

> **"이 리서치에 민감한 자료 (친구 비밀, 학교 자료, 개인 일기, 가족 사생활 등) 가 포함될 가능성이 있습니까? 있다면 `.gitignore` 에 폴더 전체를 추가해 GitHub 에 올라가지 않도록 하겠습니다."**

### 판단 기준

- **민감 O** → `.gitignore` 에 폴더 전체 등록 + 사용자 확정 후 폴더 생성
- **민감 X** → 일반 폴더로 생성
- **애매함** → "민감 O" 쪽으로 판단 (보수적)

### 민감 리서치 예시 (본인 컨텍스트에 맞게 조정)

- 친구 비밀·개인 사생활 정보
- 학교 자료 (성적, 교사 평가, 학급 정보 등)
- 개인 일기·비밀 메모
- 가족 재무·건강 기록
- 진행 중인 소송·자백 자료

### 비민감 리서치 예시

- 관심 주제 조사 (공개 자료 기반)
- 시장 조사·제품 리서치
- 공개된 정보 정리
- 취미 관련 조사

### 새 리서치 폴더 만들 때 자동 생성 서브폴더

새 리서치 폴더를 만들 때는 그 안에 다음 서브폴더도 함께 자동으로 만든다.

- `documents/` — 사용자가 리서치에 참고할 자료 (PDF, 스크린샷, 문서 등) 를 넣는 곳. Claude 가 리서치 결과에 맞춰 다운로드하는 이미지 (게임 커버, 상품 사진 등) 도 함께 이 폴더에 저장 (자녀 인지 부담 축소 목적, 폴더 하나로 통합).

리포트 markdown 에서 이미지 참조 경로는 `![alt](documents/file.png)` 형태.

## 🔒 리서치 트리거 표현 구분 (필수)

"리서치" 관련 표현은 두 단계로 구분해서 처리한다. 준비 단계와 실행 단계를 반드시 나눠서 진행 — 준비 단계에서 실제 리서치를 즉시 시작해버리는 사고를 방지하는 것이 본 룰의 핵심 목적이다.

### 준비 단계 트리거

"리서치 준비", "리서치 준비해줘" 같은 표현은 **폴더 생성 + 인터뷰 + 자료·지시 대기까지만** 진행. 실제 리서치 (웹 검색, 조사, 리포트 작성) 는 절대 시작 금지.

절차:

1. **주제 확인** — 사용자가 주제를 함께 말했으면 다음 단계로 진행. 주제가 빠져 있으면 "어떤 주제로 준비할까요?" 짧게 확인.

2. **신규 리서치 폴더 생성** — § 신규 리서치 폴더 생성 규칙 (민감 자료 여부 인터뷰 포함) 따라 폴더 생성.

3. **`documents/` 서브폴더 자동 생성**.

4. **비밀번호 인터뷰** — § 리서치 시작 시 비밀번호 인터뷰 따라 진행.

5. **사용자에게 다음 두 가지를 준비하도록 안내**:

   - `documents/` 폴더에 리서치에 참고할 자료 (PDF, 스크린샷, 문서 등) 를 넣기.
   - 리서치가 무엇을 조사할지 구체적인 지시 준비 (예: "이 자료 기반으로 내가 좋아할 만한 게임 10개 추천 리포트 만들어줘, 각 게임 옆에 리뷰 점수와 상세 페이지 링크 붙여줘" 등).

6. **사용자 답변 대기** — 자료 준비 완료 + 구체 지시가 들어오면 실행 단계로 전환.

### 실행 단계 트리거

"리서치 실행해줘", "리포트 만들어줘", "documents 폴더 파일 분석해서 리포트 만들어줘" 같은 표현은 준비 단계가 완료된 상태 (`documents/` 에 자료 있음) 를 전제로 실제 리서치 진행. 자료 분석 → 리포트 작성 → HTML 빌드 → 배포까지 자동 흐름.

### 애매한 표현 처리

"리서치 시작해줘" 처럼 준비인지 실행인지 애매한 표현을 받으면 반드시 사용자에게 확인: "폴더·인터뷰까지 준비 단계인가요, 실제 리서치 실행 단계인가요?"

## 🔒 리서치 시작 시 비밀번호 인터뷰 (필수)

새 리서치를 시작할 때 폴더 생성 직후 사용자에게 짧게 물음:

> **"이 리포트는 비밀번호 걸까요? (Y/N)"**

### N 답변

그대로 진행. 리포트는 공개, URL 아는 누구나 접근 가능.

### Y 답변

"어떤 숫자 4자리로 비번을 정할까요?" 로 이어짐 → 사용자가 그 자리에서 숫자 4개 입력.

### 실제 저장 방식

비번의 SHA-256 hash 만 리포트 HTML 에 저장하고 원 비번은 어디에도 저장 안 함. 리포트 접근 시 JavaScript 로 입력받은 숫자 4자리를 hash 해서 저장된 hash 와 비교. 심리적 장벽 수준의 방어 (완전한 보안 아님, 하지만 우연한 노출은 원천 차단).

### 비번 입력 UI

비공개 리포트 접근 시 숫자 4자리 입력 UI 를 표시. PIN 스타일의 4개 입력 박스 (한 자리씩 입력, 다음 박스로 자동 이동) 로 스마트폰에서도 편리하게 입력 가능. 잘못된 비번 입력 시 짧은 shake 애니메이션으로 알림.

리포트 목록 페이지 (`research/index.html`) 에서는 비공개 리포트도 제목 표시하되 클릭 시 비번 페이지로 이동.

## 🖼️ 이미지 자동 다운로드 & 리사이즈 (필수)

리포트에 관련 이미지가 필요할 때 (예: 게임 커버, 상품 사진, 인물 사진, 지도 등) Claude 가 자동으로 처리한다.

### 절차

1. 공식 사이트, Wikipedia, 저작권 명시된 이미지 소스에서 이미지 URL 찾기
2. `curl` 로 다운로드 (Mac/Linux) 또는 PowerShell `Invoke-WebRequest` (Windows)
3. 리사이즈:
   - Mac 은 `sips -Z 640 input.png --out output.png` (UI 스크린샷·아이콘은 PNG 유지)
   - Mac 사진 (풍경, 인물, 상품 등) 은 `sips -Z 640 -s format jpeg -s formatOptions 85 input.png --out output.jpg` 로 JPG 변환 (파일 크기 크게 감소)
   - Windows 는 PowerShell 또는 다른 이미지 처리 도구 사용
4. 리서치 폴더 안 `documents/` 에 저장
5. Markdown 리포트에 `![alt](documents/file.png)` 로 삽입 (파일 확장자 리사이즈 결과에 맞춰)
6. 리포트 하단에 저작권 안내 한 줄 자동 추가 — 예: "이미지 출처: 각 게임의 공식 사이트. 개인 리서치 목적 사용, 상업 사용 금지."

### 이미지 크기 표준

- **width 640px** — retina 2x 대비 표시 폭 320px 가 표준
- **JPG quality 85** — 사진 파일 크기 대폭 감소 (14 MB → 1 MB 흔함)
- 원본 그대로 업로드 절대 금지 (페이지 로딩 심각히 느려짐)

## 🎨 리포트 스타일 자산 참조 (필수)

리서치 리포트 HTML 을 빌드할 때 다음 자산을 참조한다. 자산 파일은 이 저장소의 `research/` 서브폴더에 있고, Render 정적 배포 시 사이트 URL 하위 경로 (`/research/...`) 로 접근 가능하다.

### 라이트 테마 CSS (필수)

리포트 HTML `<head>` 안에 다음 링크 삽입:

```html
<link rel="stylesheet" href="/research/report-style.css">
```

### 이미지 라이트박스 스크립트

리포트에 이미지 갤러리가 있으면 `</body>` 직전에 다음 스크립트 삽입 (이미지 클릭 시 확대 표시):

```html
<script src="/research/lightbox-init.js"></script>
```

### 경로 룰

- 절대 경로 `/research/...` 기준으로만 참조한다. 상대 경로 (`../report-style.css` 등) 는 사용 금지 — 리서치 폴더 위치나 슬러그가 바뀔 때 링크가 깨진다.
- Render 정적 배포에서 사이트 루트 도메인 하위 `/research/` 경로가 그대로 서빙되니 이 절대 경로 방식이 안정적이다.

## 🔒 리포트 수정 규칙 (필수)

배포된 리포트에 대한 수정 요청을 받으면 다음 흐름을 준수한다. 덕지덕지 붙여서 완성도가 떨어지는 상황을 원천 차단하는 것이 핵심 목적.

### 1. 수정 방향 사용자 컨펌 먼저

수정 요청을 받아도 바로 편집에 들어가지 않는다. 어떻게 수정할지 사용자에게 Before / After 형식으로 먼저 알리고 명시 컨펌 ("그래", "진행해", "OK" 등) 을 받은 후에만 편집.

### 2. 큰 재작성 vs 작은 수정 판단

다음 중 하나라도 해당하면 큰 재작성으로 분류하고 사용자에게 별도 확인.

- 한 섹션의 본문이 50% 이상 다시 쓰여야 할 때
- 여러 섹션의 cross-reference (§ 번호, 용어사전, 사실 일관성) 가 깨질 만큼 큰 구조 변경일 때

작은 수정으로 가장해서 큰 재작성을 계속 붙이는 안티패턴을 방지. 큰 재작성이 필요한데 계속 부분 수정으로 처리하면 최종 리포트 완성도가 크게 떨어짐.

### 3. Cross-reference 정합성 자동 점검

수정 후 다음 위치들의 정합성을 자동 점검한다.

- 변경한 표현·숫자·용어가 다른 위치에도 분산되어 있는지 grep 으로 확인
- Dual-language 리포트면 다른 언어 버전과 구조·내용 1:1 매칭 유지
- 목록 페이지 (`research/index.html`) 에 표시되는 제목·요약 정보와 리포트 본문 정합성

발견된 불일치는 일괄 정정. 한 곳만 고치고 나머지는 방치하는 방식 금지.

### 4. 재빌드 & 재배포

수정 완료 후 자동으로 HTML 재빌드 → GitHub commit + push → Render 자동 재배포. URL 은 그대로 유지되므로 이전에 공유된 링크가 자동으로 최신 버전 반영.

## Workflow Rules

### 1. 승인 없이 구현 금지

- 사용자가 문제 또는 요구사항을 설명하면 **바로 구현하지 않는다**.
- 항상 "구현할까요?" 묻고 명시 승인 후 코드 작성.

### 2. Plan 먼저 (plan.md)

- 구현 전, `plan.md` 파일에 plan 작성해서 사용자가 검토할 수 있게 한다.
- 사용자 승인 없이 코딩 시작 금지.

### 3. 인터뷰 먼저 (필요 시)

- `plan.md` 작성 전 명확화 필요 시 인터뷰 먼저.
- 인터뷰 룰:
  - 뻔하지 않고 심도 있는 질문
  - 객관식 4-10개
  - 시작 시 전체 질문 수 명시
  - 추천 옵션은 1번 배치
  - 한 번에 한 질문, 답 받은 후 다음

### 4. Research 먼저 확인 (필요 시)

- `research.md` 파일이 있으면 `plan.md` 작성 전 완전히 읽고 이해.

### 5. 배포 전 테스트 필수

- 구현 완료 후 **반드시 테스트 케이스 작성 + 테스트 실행** 후 배포.
- 이 단계 절대 skip 금지.

### 6. 실행 순서

```
Interview (필요 시) → research.md review (있으면) → plan.md → 사용자 승인 → Implementation → Test → Deploy
```
